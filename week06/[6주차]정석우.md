# 필수

## 프레임워크란?
프레임워크는 소프트웨어 개발에서 기본적인 뼈대나 구조를 제공하여, 개발자가 코드 작성 시 특정 규칙과 패턴에 따라 효율적으로 애플리케이션을 구축할 수 있도록 돕는 플랫폼이다. 프레임워크는 미리 정의된 여러 클래스, 인터페이스, 함수, 라이브러리 등을 포함하여 반복적이고 공통적인 작업을 쉽게 처리할 수 있게 해준다. 이는 자주 사용하는 기능들을 미리 정의해 두고 필요 시 재사용할 수 있는 코드 재사용성과 표준화를 통한 일관된 코딩 방식과 구조를 제공하여 코드의 유지보수를 높이는 장점이 있다. 우리가 학습하는 스프링(Spring) 또한 자바에서 사용하는 프레임워크의 종류이며, 파이썬에서 사용하는 Django, 자바스크립트에서 사용하는 React 등이 있다. 여러 종류의 프레임워크가 존재하는 만큼 개발자는 본인의 개발 방향에 맞게 알맞은 프레임워크를 선택하는 것이 개발 과정에서 중요한 역할을 한다.
또한 프레임워크를 언급한다면 라이브러리를 빼놓을 수 없다. 라이브러리는 응용 프로그램 개발을 위해 필요한 기능을 모아놓은 소프트웨어로 빈번하게 사용되는 리소스를 모아둔 것이다. 라이브러리에는 미리 작성된 코드, 변수, 함수, 클래스 등이 포함될 수 있고 필요한 곳에서 호출하여 사용할 수 있다. 프레임워크와 라이브러리의 차이점을 명확하게 분리하여 정의하는 것은 어렵다. 다만 프레임워크는 그 스스로 제어 흐름의 주도성을 갖는 반면, 라이브러리는 단순 개발 툴의 역할을 하기 때문에 흐름의 주도성을 개발자가 갖고 사용한다.

## 스프링의 장점 / 사용이유
스프링 프레임워크는 Java 기반의 애플리케이션 개발을 효율적이고 체계적으로 돕기 위해 설계된 강력한 프레임워크이다. 주로 대규모 엔터프라이즈 애플리케이션에서 많이 사용되며, 개발 생산성과 유지보수성을 크게 높여주는 장점이 있다.
스프링의 장점
1.	의존성 주입(DI)과 제어의 역전(IoC) 지원<br/>
      o	스프링은 DI와 IoC 컨테이너를 제공하여 객체 간의 의존성을 외부에서 주입하고, 객체 생성과 관리를 스프링이 담당한다. 이로 인해 코드 간 결합도가 낮아지고 확장성과 테스트가 용이해진다.
2.	모듈화된 구조<br/>
      o	스프링은 여러 모듈로 구성되어 있어, 필요에 따라 기능을 추가할 수 있다. 개발자는 필요한 모듈만 사용해 애플리케이션을 구성할 수 있다.
3.	트랜잭션 관리<br/>
      o	스프링은 선언적 트랜잭션 관리를 제공하여, 복잡한 비즈니스 로직을 구현할 때 데이터의 일관성과 안정성을 보장한다. 트랜잭션 처리를 코드에서 직접 작성할 필요 없이 설정만으로 관리할 수 있다.
4.	POJO 기반의 경량 프레임워크<br/>
      o	스프링은 POJO(Plain Old Java Object)를 기반으로 하여 특정 프레임워크에 종속되지 않고 유연하게 설계할 수 있다. 이는 서버나 특정 환경에 의존하지 않게 만들어 다양한 환경에서 유연하게 동작한다.

스프링은 이러한 장점을 기반으로 복잡한 비즈니스 로직 관리, 유연한 확장성을 제공한다. 이는 대규모 Java 애플리케이션 개발 시 개발 생산성을 높여주기 때문에 많은 개발자들이 사용한다.

## AOP란?
AOP(Aspect-Oriented Programming, 관점 지향 프로그래밍)은 핵심 비즈니스 로직과는 별개인 공통 기능을 분리하여 모듈화하는 프로그래밍 기법이다. 스프링에서 AOP는 애플리케이션의 핵심 로직을 간결하게 유지하면서 횡단 관심사(cross-cutting concerns)를 별도로 관리할 수 있게 돕는다. 이로 인해 코드 중복을 줄이고, 유지보수를 더 쉽게 할 수 있다.
AOP의 주요 개념
1.	Aspect: 횡단 관심사를 모듈화한 것으로, 예를 들어 로깅이나 트랜잭션 관리 등의 기능을 Aspect로 분리한다.
2.	Advice: Aspect가 실행할 실제 동작이나 작업을 정의한다. 특정 시점에 수행할 동작을 설정할 수 있다.
3.	Join Point: Advice가 적용될 수 있는 지점으로, 보통 메서드 실행 지점이 Join Point가 된다.
4.	Pointcut: Advice가 적용될 구체적인 지점을 정의한 것으로, 특정 메서드나 클래스에 대해 설정할 수 있다.
      Ex) 애플리케이션 전반에 걸쳐 로깅을 해야 하는 경우 AOP를 사용하면, 로깅이 필요한 부분에만 적용하는 코드를 작성하지 않고, 핵심 비즈니스 로직 외부에서 일관되게 관리할 수 있다.

## 싱글톤이란?
싱글톤 패턴은 클래스의 인스턴스가 오직 하나만 생성되도록 보장하는 디자인 패턴이다. 어플리케이션 전체에서 동일한 인스턴스를 공유하여 메모리 낭비를 줄이고, 특정 객체가 전역적으로 하나만 필요할 때 사용한다.

---

## 싱글톤 컨테이너란?
싱글톤 컨테이너는 애플리케이션 내에서 여러 객체를 싱글톤 패턴으로 관리하고, 필요할 때 해당 객체들을 제공해주는 객체 관리 시스템이다. 주로 스프링 프레임워크와 같은 IoC(Inversion of Control) 컨테이너에서 사용되며, 애플리케이션 전역에서 동일한 객체 인스턴스를 한 번만 생성해 공유함으로써 메모리 효율성과 성능을 높인다.
싱글톤 컨테이너의 핵심 역할은 객체의 싱글톤 패턴 관리, 전역 접근 제공, 그리고 의존성 주입 기능이다. 컨테이너는 애플리케이션에서 필요로 하는 객체를 하나씩만 생성해 동일한 인스턴스를 유지하여 관리한다. 이를 통해 애플리케이션 내에서 반복적으로 객체를 생성하지 않아도 되며, 메모리 사용을 최적화할 수 있다. 또한, 싱글톤 컨테이너는 애플리케이션 어디서든 동일한 객체에 접근할 수 있도록 전역 접근을 제공하여, 객체 생성을 중앙에서 제어하고 관리할 수 있게 한다.
특히 싱글톤 컨테이너는 객체 간의 의존성 주입을 통해 결합도를 낮추는 역할을 한다. 예를 들어, 하나의 객체가 다른 객체를 필요로 할 때, 싱글톤 컨테이너가 해당 객체에 의존성을 주입해주므로, 각 객체가 서로에게 강하게 의존하지 않고 독립적으로 개발될 수 있다. 이로 인해 코드의 재사용성과 유지보수성이 높아지며, 애플리케이션의 확장이 용이해진다.
스프링 프레임워크의 IoC 컨테이너가 싱글톤 컨테이너의 대표적인 예로, 스프링은 기본적으로 모든 스프링 빈(Bean)을 싱글톤 패턴으로 관리한다. 개발자가 특별히 설정을 하지 않는 한 스프링 컨테이너는 각 빈을 하나의 인스턴스로만 생성하여 애플리케이션 전역에서 사용하도록 보장한다. 이를 통해 개발자는 객체 생성과 관리를 스프링에 맡기고, 필요한 경우 컨테이너에서 객체를 요청하여 사용하면 된다.

## Spring에서 Bean 관리 방법
Spring에서 Bean 관리 방법은 IoC 컨테이너를 통해 이루어지며, 주로 XML 설정, Java 설정, 그리고 애너테이션을 사용하여 관리할 수 있다. Spring은 싱글톤 패턴을 기본으로 Bean을 생성하고 관리하며, 객체의 생명주기를 포함해 의존성을 관리하고 필요에 따라 주입한다.
1. XML 설정을 통한 Bean 관리<br/>
   스프링의 전통적인 방식으로, applicationContext.xml과 같은 XML 파일을 사용해 Bean을 정의하고, 속성을 설정한다. XML 설정은 필요한 Bean의 이름, 클래스, 속성 등을 명시적으로 설정할 수 있어 가독성이 좋지만, 코드와 설정이 분리되어 있어 복잡한 설정에서는 관리가 어려울 수 있다.

2. Java 설정을 통한 Bean 관리<br/>
   Java 설정 방식은 @Configuration과 @Bean 애너테이션을 사용하여 Java 클래스에서 직접 Bean을 정의하고 관리하는 방식이다. 코드와 설정이 한 곳에 있어 유지보수가 용이하고, 타입 안전성을 제공하며 IDE의 코드 자동 완성 기능을 활용할 수 있다.

3. 애너테이션을 통한 Bean 관리<br/>
   스프링에서는 클래스에 직접 애너테이션을 붙여 Bean을 자동으로 등록하고 관리할 수 있다. @Component, @Service, @Repository, @Controller 같은 애너테이션을 사용하여 클래스에 붙이면 스프링이 자동으로 Bean으로 인식하여 IoC 컨테이너에 등록한다. @ComponentScan을 통해 특정 패키지를 스캔하여 자동으로 Bean을 등록할 수 있다.

4. 스코프 설정을 통한 Bean 범위 관리<br/>
   스프링에서는 Bean의 스코프(scope)를 설정하여 Bean이 생성되고 관리되는 방식을 정의할 수 있다. 기본적으로 스프링은 모든 Bean을 싱글톤(singleton) 스코프로 관리하지만, 필요에 따라 다른 스코프를 사용할 수 있다.<br/>
   주요 스코프<br/>
   •	singleton: 스프링 IoC 컨테이너에서 하나의 인스턴스만 생성하고 재사용한다.<br/>
   •	prototype: 요청할 때마다 새로운 인스턴스를 생성한다.<br/>
   •	request: HTTP 요청 당 하나의 Bean 인스턴스를 생성한다(웹 애플리케이션에서 사용).<br/>
   •	session: HTTP 세션 당 하나의 Bean 인스턴스를 생성한다(웹 애플리케이션에서 사용).<br/>

5. 의존성 주입 (Dependency Injection, DI)<br/>
   스프링은 의존성 주입을 통해 Bean 간의 관계를 설정하고, 자동으로 주입할 수 있도록 한다. 의존성 주입 방식에는 생성자 주입, 필드 주입, Setter 주입이 있다.<br/>
   •	@Autowired: 의존성 주입을 자동으로 수행하며, 필요한 Bean을 IoC 컨테이너에서 찾아서 주입한다.<br/>
   •	@Qualifier: 같은 타입의 여러 Bean이 있을 경우 특정 Bean을 주입하도록 지정한다.<br/>
   •	@Primary: 여러 Bean 중 기본으로 주입할 Bean을 지정할 수 있다.<br/>

## Layered Architecture
Layered Architecture(계층형 아키텍처)는 소프트웨어 시스템을 계층으로 나누어 각 계층이 특정 역할과 책임을 갖도록 설계하는 아키텍처 패턴이다. 일반적으로 프레젠테이션, 비즈니스 로직, 데이터 접근 계층으로 나누어지며, 각 계층은 명확한 역할을 가지고 계층 간의 결합을 줄여 애플리케이션의 유지보수성과 확장성을 높인다.<br/>
주요 계층 구성<br/>
1.	프레젠테이션 계층 (Presentation Layer): 사용자 인터페이스나 API 엔드포인트를 담당하는 계층으로, 사용자 입력을 받고 응답을 제공한다. 이 계층은 사용자로부터 입력을 받아 비즈니스 계층에 전달하며, 처리 결과를 사용자에게 반환하는 역할을 한다.
2.	비즈니스 계층 (Business Logic Layer): 애플리케이션의 핵심 로직을 담당하는 계층으로, 비즈니스 규칙과 절차를 처리한다. 이 계층은 프레젠테이션 계층에서 요청받은 작업을 수행하며, 데이터 계층과 상호작용해 필요한 데이터를 처리한다. 비즈니스 로직 계층은 주로 서비스와 도메인 모델을 포함한다.
3.	데이터 접근 계층 (Data Access Layer): 데이터베이스와의 상호작용을 담당하는 계층으로, 데이터의 저장, 수정, 삭제, 조회와 같은 작업을 처리한다. 데이터베이스를 직접 다루는 대신, SQL 쿼리나 ORM을 통해 데이터를 관리하며, 이를 통해 비즈니스 계층과 데이터베이스 간의 결합을 줄인다.
4.	공통 계층 (Common Layer) (선택 사항): 여러 계층에서 사용할 수 있는 공통 기능(예: 로깅, 보안)을 제공한다. 각 계층에 걸쳐 재사용할 수 있는 유틸리티나 서비스들을 포함한다.
      Layered Architecture의 장점
      •	유지보수 용이성: 각 계층이 독립적으로 설계되어 있어, 특정 계층을 수정해도 다른 계층에 영향을 주지 않기 때문에 유지보수가 쉽다.
      •	확장성: 계층별로 새로운 기능을 추가하거나 기존 기능을 확장하기 쉽다.
      •	결합도 감소: 각 계층이 독립적이므로 코드의 결합도가 낮아지며, 각 계층이 명확한 역할을 가지므로 코드가 이해하기 쉬워진다.
      •	테스트 용이성: 각 계층이 독립적이기 때문에 계층별로 테스트가 가능하다. 특히 비즈니스 로직 계층을 단위 테스트하기가 용이하다.

## 스프링 어노테이션란? / 어노테이션 종류 10가지 이상 찾아보기
스프링 애너테이션(Spring Annotation)은 스프링 프레임워크에서 제공하는 특수한 메타데이터로, 클래스, 메서드, 필드에 붙여 특정 기능을 수행하거나 설정할 수 있도록 돕는 역할을 한다. 애너테이션은 XML 설정을 대체하거나 보완하여 코드를 더 간결하고 직관적으로 작성할 수 있게 한다.
1. @Component<br/>
   •	기본적으로 스프링이 관리하는 Bean임을 나타낸다. 클래스에 붙여 객체를 자동으로 생성하고, IoC 컨테이너에 등록한다.
2. @Service<br/>
   •	비즈니스 로직을 수행하는 서비스 클래스에 주로 붙이며, 스프링이 관리하는 Bean임을 나타낸다. @Component와 유사하지만, 역할을 명확히 하기 위해 서비스 레이어에 사용된다.
3. @Repository<br/>
   •	데이터베이스 접근 계층에서 사용하며, 데이터 액세스 예외(transactions)을 스프링 예외로 자동 변환한다. DAO 클래스에 주로 사용된다.
4. @Controller<br/>
   •	웹 애플리케이션에서 사용하며, 클라이언트의 요청을 처리하고 적절한 응답을 반환하는 클래스에 붙인다. 주로 프레젠테이션 계층에서 사용된다.
5. @RestController<br/>
   •	@Controller와 @ResponseBody를 결합한 애너테이션으로, RESTful 웹 서비스를 제공하는 컨트롤러에서 사용된다. 모든 메서드가 JSON 또는 XML 형식으로 데이터를 반환하게 된다.
6. @Autowired<br/>
   •	의존성 주입을 위해 사용된다. 생성자, 필드, 메서드에 붙일 수 있으며, 스프링이 IoC 컨테이너에서 해당 타입의 Bean을 자동으로 주입해준다.
7. @Qualifier<br/>
   •	@Autowired와 함께 사용되며, Bean 주입 시 같은 타입의 여러 Bean 중 특정 이름을 가진 Bean을 선택할 때 사용한다.
8. @Configuration<br/>
   •	설정 정보를 담은 클래스를 나타내며, 스프링 컨테이너가 해당 클래스를 설정 파일로 인식하도록 한다. 클래스 내에서 Bean을 정의할 때 사용된다.
9. @Bean<br/>
   •	@Configuration 클래스에서 사용하여 스프링 IoC 컨테이너에 Bean을 등록하는 메서드를 나타낸다.
10. @Primary<br/>
    •	같은 타입의 Bean이 여러 개 있을 때, 특정 Bean을 기본으로 주입하도록 지정한다. @Autowired와 함께 사용되며, 주입 우선순위를 설정한다.
